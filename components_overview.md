# System Architecture & Components Overview

This document provides a detailed breakdown of the different components in the Multi-Agent System (MAS), how they are structured, and how they connect to one another.

## 1. Graph Module (`graph/`)
The graph is the backbone of the system. It controls the flow of data and determines which agent runs at what time. It is built using **LangGraph**.

### `graph/state.py` (The Memory)
- **Role**: Defines the `GraphState` TypedDict. This acts as the shared memory or "blackboard" that all agents can read from and write to as the workflow executes.
- **Connections**: It is passed from node to node. Every agent and module receives this state, parses it (e.g., extracts `current_event` or `factory_context`), and returns a dictionary to update it (e.g., appending to `agent_proposals`).

### `graph/workflow.py` (The Routing Engine)
- **Role**: Defines the execution sequence (`StateGraph`). It instantiates the singleton instances of all agents and the RAG retriever, wrapping them in node functions.
- **Connections**: 
  - It creates a strict linear pipeline: `RAG Node -> Machine Agent -> Line Agent -> Scheduler Agent -> END`.
  - It handles the transition of the `GraphState` from one function to the next.

## 2. RAG Module (`rag/factory_data.py`)
- **Role**: Simulates a Retrieval-Augmented Generation (RAG) system to provide factory operational data.
- **Mechanism**: 
  - Loads data from `data/factory_logs.txt` using LangChain's `TextLoader`.
  - Chunks the data and embeds it into a local `Chroma` vector database using `OllamaEmbeddings` (model: `nomic-embed-text`).
- **Connections**: Driven by the `retrieve_context_node` in the graph. It uses the `current_event` to query the vector database and injects the results into the `GraphState` (`factory_context`), allowing downstream agents to ground their responses in actual factory data.

## 3. Agents Module (`agents/`)
The decision-making brains of the operation. Each agent serves a specific hierarchical role, built using the `ChatOllama` LLM (model: `llama3.2`) and LangChain prompt templates.

### `agents/machine_agent.py`
- **Role**: The lowest-level autonomous agent representing a single machine on the factory floor.
- **Function**: Focuses narrowly on machine-level constraints like breakdown assessments, part availability, or local repair times.
- **Connections**: Receives the `current_event` and `factory_context`. Its output is appended to the `agent_proposals` array in the shared state, feeding directly into the higher-level agents.

### `agents/production_line_agent.py`
- **Role**: The mid-level agent representing an entire production line or cell.
- **Function**: Synthesizes the narrow assessments made by the Machine Agent(s) and figures out line-wide adaptations. For example, if Machine D fails, it decides if the line should route parts to Machine E or slow down the conveyor.
- **Connections**: Reads the `current_event`, `factory_context`, AND the `agent_proposals` generated by the Machine Agent. It adds its own broader proposal to the `agent_proposals` list for the Scheduler to see.

### `agents/scheduler_agent.py`
- **Role**: The top-level orchestrator.
- **Function**: Responsible for the global factory schedule. It takes all the fragmented proposals from the lower agents and merges them into a single, cohesive, authoritative plan.
- **Connections**: Reads everythingŌĆöthe `current_event`, `factory_context`, the combined `agent_proposals` from both subordinate agents, and the current global `schedule`. It overwrites the `schedule` in the `GraphState` and finalizes the run.

## Summary of the Data Flow:
1. `Event` -> **RAG** -> outputs `Context`.
2. `Event` + `Context` = **Machine Agent** -> outputs `Machine Proposal`.
3. `Event` + `Context` + `Machine Proposal` = **Production Line Agent** -> outputs `Line Proposal`.
4. `Event` + `Context` + `Machine Proposal` + `Line Proposal` + `Old Schedule` = **Scheduler Agent** -> outputs `New Schedule`.
